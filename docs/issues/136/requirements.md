# Requirements - Issue #136: パイプライン管理用プライベートリポジトリの構築

## ユーザーストーリー

「自動化ツールの管理者として、Spotify の番組 ID や URL などの機密性の高い設定情報、および本番運用のためのパイプライン定義を、パブリックリポジトリから分離して安全に管理したい。また、パブリック側のコード更新を検知して、セルフホストランナー上で安全に処理を実行できる仕組みを利用したい。」

## 背景
`my-apps` はパブリックリポジトリであるため、個別の番組設定や本番用ワークフローを置くことができない。セキュリティリスクを最小化するため、設定とオーケストレーションのみを担当するプライベートリポジトリ `private-ops` を構築する。

## 受け入れ基準 (Acceptance Criteria)

### 1. プライベートリポジトリの構造
- [ ] 設定ファイルを管理する `configs/` ディレクトリが存在すること。
- [ ] ワークフローを管理する `.github/workflows/` ディレクトリが存在すること。

### 2. 設定ファイルのフォーマット
- [ ] 番組名、番組ID (Show ID)、ベースURL、エピソード管理情報などを含む JSON フォーマットが定義されていること。
- [ ] 複数の番組（テスト用、本番用A、本番用Bなど）を切り替え可能な構造であること。

### 3. クロスリポジトリ連携
- [ ] `my-apps` から `private-ops` のワークフローをトリガーできる `repository_dispatch` の設定（PAT含む）が検討されていること。
- [ ] `private-ops` のジョブが `runs-on: self-hosted` で実行されるように定義されていること。

### 4. セキュリティ
- [ ] パブリックリポジトリには一切の機密情報が含まれない状態を維持すること。
